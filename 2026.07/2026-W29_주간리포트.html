덕파

Opus 4.7 토큰 부담과 Codex의 기능 추격 속, Claude Code와 Codex는 우열이 아니라 용도로 갈린다

Opus 4.7의 토크나이저 변경으로 토큰 소모가 늘고 Claude Code에 연이은 사고가 겹치면서, GPT-5.5를 얹은 Codex가 빈틈을 파고들었습니다. 성능·한도·컨텍스트·기업 철학까지 뜯어본 결과 어느 한쪽의 압도적 우위는 없고, 작업 성격에 따라 도구를 나눠 쓰는 것이 결론입니다.

핵심 포인트

  • Opus 4.7 새 토크나이저는 외부 측정 기준 1.20~1.47배 더 많은 토큰을 쓰는 반면, GPT-5.5는 Codex 작업에서 약 40% 토큰 효율을 발표
  • AMD 스텔라 로렌조의 실측(6,852세션·234,760회 도구 호출)에서 Opus 4.7 이후 사고 깊이 중앙값이 67% 하락
  • SWE-bench Pro(64.3% 대 57.7%)·SWE-bench Verified(87.6% 대 74.9%)에서는 Opus 계열이 앞섬
  • Claude Code는 1M 컨텍스트를 추가 요금 없이 정식 지원, Codex는 400K이며 1M은 opt-in 단계
  • Anthropic은 '함께 가는 동료(companion)', OpenAI는 '위임받는 작업자(delegated worker)'라는 도구 철학 차이가 UX 설계에 그대로 반영

왜 지금

코딩 에이전트 시장의 무게중심이 Claude Code 독주에서 Codex와의 양강 구도로 옮겨간 흐름을 정량 데이터로 포착한 글로, 3주 연속 인기 1위를 지키며 이 논쟁이 여전히 중심임을 보여줍니다.

실무 시사점

장기·다중 파일 리팩터링이나 UI 마감이 잦은 팀은 Claude Code를, 명확한 스펙 기반 일상 코딩과 토큰 효율이 중요한 팀은 Codex를 우선 검토하고, 여유가 있다면 두 도구를 작업 유형별로 병행하는 전략이 현실적입니다.

실무 적용 포인트

  • 장기 리팩터링·1M 컨텍스트가 필요한 작업은 Claude Code, 스펙이 명확한 일상 코딩·토큰 절약이 중요하면 Codex로 작업을 나눠 배분해본다
  • 팀 내 코딩 에이전트 토큰 소모량을 주기적으로 측정해, 모델 업데이트 이후 체감 성능 변화를 데이터로 확인하는 습관을 들인다

핵심 용어

토크나이저(tokenizer) 텍스트를 모델이 처리하는 토큰 단위로 쪼개는 방식으로, 바뀌면 같은 글도 토큰 소모량이 달라짐SWE-bench 실제 오픈소스 저장소의 버그 수정 능력을 측정하는 코딩 모델 벤치마크컨텍스트 윈도우 모델이 한 번에 참고할 수 있는 텍스트(코드·대화)의 최대 범위

핵심 포인트

  • 에이전트를 가르는 결정적 축은 '권한(어디까지 알아서 하게 둘 것인가)'이며, 웹 에이전트 < 코딩 에이전트/컴퓨터 유즈 < 자율 에이전트 순으로 자유도가 커짐
  • Claude Code는 2025년부터 운영된 기준점 도구로, Sonnet 4.6·Opus 4.8(2026년 5월 28일 출시) 기반이며 1M 컨텍스트를 정식 지원

  • Claude Cowork는 비개발자용 GUI 버전으로, 'Ask before acting' 방식으로 실행 전 계획을 보여주고 승인을 기다림
  • 코딩 에이전트는 이제 코드뿐 아니라 파일 정리·문서 자동화·앱 실행 등 컴퓨터의 거의 모든 작업으로 영역이 확장됨

실무 시사점

  • 비개발자 조직이라면 코딩 에이전트 대신 Claude Cowork 같은 GUI 기반 대안부터 시범 도입한다

덕파

OpenRouter 토큰 1위를 탈환한 오픈소스 에이전트 Hermes — '쓸수록 나에게 맞춰지는' self-improving 구조

Nous Research가 MIT 라이선스로 공개한 오픈소스 에이전트 Hermes Agent가 출시 두 달 반 만에 OpenRouter 실사용 토큰 1위(누적 1조200억 토큰, GitHub 스타 14.5만)에 올랐습니다. 핵심은 'self-improving' — 성공한 작업 절차를 스킬로 자동 저장하고, 검색·수정하며, 사용자 자체를 모델링해 세션을 넘어 맞춰집니다. 설치·설정 마법사와 OpenClaw 마이그레이션까지 정리했습니다.

핵심 포인트

  • OpenRouter 실사용 토큰 랭킹에서 기존 1위 OpenClaw를 제치고 1위 탈환(2026-05-10) — 누적 1조200억 토큰, 출시 11주 GitHub 스타 14.5만
  • self-improving 4대 메커니즘: ①도구 5회+ 성공 작업 절차를 마크다운 스킬로 자동 저장 ②SQLite FTS5로 스킬 검색(벡터DB 없이 1만 문서 ~10ms) ③사용 중 스킬 자가 수정 ④Honcho로 사용자 모델링
  • 인프라가 가벼움 — 별도 벡터DB 없이 시스템 SQLite로 처리, 5달러 가상서버에서도 구동
  • Claude·OpenAI·DeepSeek·Qwen·Ollama·vLLM 등 대부분 LLM 지원(단 컨텍스트 6.4만 토큰 이상 필요), 메신저 20종+·OpenClaw 원클릭 마이그레이션
  • 한계: 스킬 포이즈닝·MCP 샌드박스 부재·자격증명 노출·GDPR 미해결 → 1인 개발자엔 적합하나 규제·감사 필요한 팀엔 부적합이라는 평

왜 지금

에이전트 서비스가 '처음 동작하는 신기함'을 넘어 '쓸수록 내 워크플로에 맞춰지는' 단계로 진입했음을 보여주는 사례로, 3767(유진 얀 5원칙)·3792(개인 LLM 위키)에서 나온 '맥락 누적' 문제의식을 제품 메커니즘으로 구현했습니다.

실무 시사점

반복 작업이 많은 1인 개발자·메이커라면 self-improving 에이전트로 절차를 자산화해볼 만하지만, 컴퓨터·계정 권한을 통째로 위임하는 특성상 보안·규제가 중요한 팀 도입은 가드레일(샌드박스·자격증명 관리)을 먼저 갖춰야 합니다.

실무 적용 포인트

  • 자주 반복하는 다단계 작업을 에이전트 스킬(마크다운 절차)로 자산화해, 도구를 바꿔도 학습이 이어지게 한다
  • 에이전트 서비스 도입 전 권한 범위·자격증명 노출·샌드박스 여부를 체크리스트로 점검한다

핵심 서비스에 특정 벤더 모델을 깊게 묶었다면, '가장 똑똑한 모델'만이 아니라 '끊기지 않는 모델'을 기준에 넣어 벤더·국가 분산과 오픈소스·온프레미스 비상 옵션을 점검할 시점입니다.

실무 적용 포인트

  • 핵심 기능에 쓰는 LLM을 벤더·국가 기준으로 이중화하고, 대체 모델로 빠르게 교체 가능한 추상화 계층을 둔다
  • 평소 안 쓰더라도 오픈소스/온프레미스 모델을 '비상 옵션'으로 확보·검증해둔다

핵심 용어

소버린 AI(Sovereign AI) 특정 국가·기업에 종속되지 않도록 자국·자체가 통제권을 갖는 AI 인프라·모델수출통제(export control) 국가안보 등을 이유로 특정 기술·제품의 해외 제공을 제한하는 규제온프레미스(on-premise) 클라우드가 아니라 자체 서버·환경에서 직접 운영하는 방식(외부 차단 위험을 줄임)

강남언니

'전 직원이 AI를 쓴 다음'이 진짜 시작 — 강남언니의 전사 AX 4단계(항구·동경·가두리·문화) 실전기

성형·시술 플랫폼 강남언니가 전사 AX(AI 전환)를 추진하며 겪은 과정을 4개 페이즈로 정리한 사례입니다. 경영진의 '투자' 선언과 비용 지원(Phase0), '저 사람도 한다' 사례로 동경 만들기(Phase1), 자율에 가드레일을 붙이는 '가두리'와 팀별 준전문가(A.I.bler) 양성(Phase2), 표준·협업·채용 기준을 다시 세우는 문화(Phase3)까지 다룹니다.

핵심 포인트

  • Phase0(항구): '완벽한 인프라'가 아니라 '움직여도 된다는 신호' — 경영진의 'AI는 투자' 공식화 + 직군 무관 도구비 지원 + 비용·안전 모니터링 책임자 지정
  • Phase1(동경): LLM 설명 대신 '내 일이 이렇게 바뀌었다'를 보여주기, 특히 비개발 동료의 결과물('저 사람도 한다')이 강력 — HR의 면접 스케줄링 자동화, 디자이너의 크롬 익스텐션 사례
  • Phase2(가두리/하네스): 자율은 가드레일이 있을 때 가속 — 권한·보안 절차, 비용 가시화(통제가 아닌 자기인식), 'MVP를 찾아주는' Productify 스킬
  • '한 사람이 전사 AI를 가르칠 수 없다' → 팀마다 준전문가 A.I.bler를 키워 자생적으로 굴러가게(엑셀 숙련자 비유)
  • Phase3(문화): 고정 표준 대신 '빠르게 합의·갈아엎는' 운영, 직무 경계를 넘나드는 새 협업 문법, 채용은 '지식'보다 '문제 정의력·다시 시도하는 태도'(지혜)

왜 지금

개인의 AI 활용을 넘어 '조직 전체가 AI로 일하게 만드는' AX가 화두가 되는 흐름에서, 도구 도입 이후의 진짜 난제(권한·비용·문화·협업)를 현장 언어로 짚은 국내 사례입니다.

실무 시사점

전사 AI 도입을 맡았다면 '도구 깔기'보다 ①움직여도 된다는 신호(경영진·비용) ②비개발 동료의 성공 사례 확산 ③가드레일 속 자율 ④팀별 준전문가 양성 순으로 설계하는 것이 실효적입니다.

실무 적용 포인트

  • AI 도입 초기엔 교육보다 '비개발 동료의 실제 결과물'을 전사에 공유해 '나도 할 수 있다'는 동경을 먼저 만든다
  • 자율이 과열되면 권한·비용 가드레일과 팀별 준전문가(A.I.bler)를 붙여 자생적 확산 구조로 전환한다

핵심 용어

AX(AI Transformation) 조직 전체가 AI를 활용해 일하는 방식으로 전환하는 것하네스/가두리(harness) 자율을 막지 않으면서 안전하게 가속하도록 두는 최소한의 가드레일·틀준전문가(A.I.bler) 각 팀에서 AI로 팀 문제를 풀어주도록 키워, 중앙팀 없이 확산되게 하는 사람
