정말로 지금은 Codex가 Claude Code보다 나을까?

덕파

Opus 4.7 토큰 부담과 Codex의 기능 추격 속, Claude Code와 Codex는 우열이 아니라 용도로 갈린다

Opus 4.7의 토크나이저 변경으로 토큰 소모가 늘고 Claude Code에 연이은 사고가 겹치면서, GPT-5.5를 얹은 Codex가 빈틈을 파고들었습니다. 성능·한도·컨텍스트·기업 철학까지 뜯어본 결과 어느 한쪽의 압도적 우위는 없고, 작업 성격에 따라 도구를 나눠 쓰는 것이 결론입니다.

핵심 포인트

  • Opus 4.7 새 토크나이저는 외부 측정 기준 1.20~1.47배 더 많은 토큰을 쓰는 반면, GPT-5.5는 Codex 작업에서 약 40% 토큰 효율을 발표
  • AMD 스텔라 로렌조의 실측(6,852세션·234,760회 도구 호출)에서 Opus 4.7 이후 사고 깊이 중앙값이 67% 하락
  • SWE-bench Pro(64.3% 대 57.7%)·SWE-bench Verified(87.6% 대 74.9%)에서는 Opus 계열이 앞섬
  • Claude Code는 1M 컨텍스트를 추가 요금 없이 정식 지원, Codex는 400K이며 1M은 opt-in 단계
  • Anthropic은 '함께 가는 동료(companion)', OpenAI는 '위임받는 작업자(delegated worker)'라는 도구 철학 차이가 UX 설계에 그대로 반영

왜 지금

코딩 에이전트 시장의 무게중심이 처음으로 Claude Code 독주에서 Codex와의 양강 구도로 옮겨가는 시점을 정량 데이터로 포착했다는 점에서, 이번 주 AI 코딩 도구 지형 변화를 보여주는 대표 사례입니다.

실무 시사점

장기·다중 파일 리팩터링이나 UI 마감이 잦은 팀은 Claude Code를, 명확한 스펙 기반 일상 코딩과 토큰 효율이 중요한 팀은 Codex를 우선 검토하고, 여유가 있다면 두 도구를 작업 유형별로 병행하는 전략이 현실적입니다.

실무 적용 포인트

  • 장기 리팩터링·1M 컨텍스트가 필요한 작업은 Claude Code, 스펙이 명확한 일상 코딩·토큰 절약이 중요하면 Codex로 작업을 나눠 배분해본다
  • 팀 내 코딩 에이전트 토큰 소모량을 주기적으로 측정해, 모델 업데이트 이후 체감 성능 변화를 데이터로 확인하는 습관을 들인다

핵심 용어

토크나이저(tokenizer) 텍스트를 모델이 처리하는 토큰 단위로 쪼개는 방식으로, 바뀌면 같은 글도 토큰 소모량이 달라짐SWE-bench 실제 오픈소스 저장소의 버그 수정 능력을 측정하는 코딩 모델 벤치마크컨텍스트 윈도우 모델이 한 번에 참고할 수 있는 텍스트(코드·대화)의 최대 범위

클로드 코드를 떠나 오픈소스로 돌아간 이유

반기황

예측 불가능한 사용량 제한에 지친 개발자가 Ollama·DeepSeek·Qwen 조합으로 워크플로를 재설계한 실전기

데모 직전 사용 한도 초과를 겪은 엔지니어가 불투명한 사용량 정책과 비용 부담을 이유로 클로드 코드를 떠나, 로컬·클라우드 오픈소스 모델로 대체하는 과정을 정리했습니다. 일상 작업의 80~90%는 오픈소스 모델로 충분하고, 정말 어려운 10~20%만 최상위 모델을 쓰는 하이브리드 전략을 제안합니다.

핵심 포인트

  • Anthropic이 사용량 제한(rate limit)을 강화하며 Pro 플랜도 하루 2~4시간 만에 한도 초과에 도달하는 사례가 다수 보고됨
  • DeepSeek Coder V2·Qwen 3 Coder가 HumanEval 등 벤치마크에서 Claude 3.5 Sonnet과 대등한 수준까지 성능 격차를 좁힘
  • Ollama로 노트북에서 7B~14B 모델 구동 가능, 32GB RAM이면 양자화로 32B급까지 가능
  • Groq·Together AI 같은 오픈소스 모델 호스팅은 Claude API 대비 약 10분의 1 비용
  • 저자의 실제 운영 방식: 일상 코딩은 로컬 Qwen 3 Coder, 복잡한 설계는 Groq의 DeepSeek V3, 정말 필요할 때만 Claude API 호출

왜 지금

Claude Code 독점 구도에 균열이 생기고 있다는 3771 기사와 같은 흐름에서, 이번엔 '가격·안정성' 관점의 실사용자 이탈 사례라는 점에서 이번 주 AI 코딩 도구 지형 변화를 보여주는 또 다른 축입니다.

실무 시사점

고정 비용으로 예측 가능한 처리량이 필요한 팀은 오픈소스 모델을 메인으로, 클로드는 복잡한 아키텍처·대규모 리팩터링 등 고난도 작업에만 선택적으로 쓰는 비용 최적화 구조를 검토할 만합니다.

실무 적용 포인트

  • 복잡도가 낮은 반복 코딩 작업부터 로컬/오픈소스 모델로 전환해보고, 실패율이 허용 범위인지 직접 측정한다
  • Groq·Together AI 같은 저비용 호스팅으로 고난도 작업만 선택적으로 상위 모델(Claude 등)에 위임하는 계층적 워크플로를 설계한다

핵심 용어

양자화(quantization) 모델 가중치의 정밀도를 낮춰 메모리·연산량을 줄이는 기법으로, 로컬 환경 구동을 가능하게 함rate limit(사용량 제한) 일정 시간 동안 사용할 수 있는 요청·토큰량을 제한하는 서비스 정책Ollama 오픈소스 LLM을 로컬 컴퓨터에서 손쉽게 내려받아 구동할 수 있게 해주는 도구

'에이전트'라는 이름이 남발되는 혼란 속에서, 무엇을 알고(컨텍스트)·무엇으로 일하고(도구)·어디까지 맡기고(권한)·언제 켜지는지(트리거)라는 네 축으로 서비스를 분류했습니다. 1편에서는 가볍게 맡기는 웹 에이전트(Manus·Genspark)와 매번 승인이 필요한 코딩 에이전트(Claude Code·Codex·Antigravity·Cowork) 6개를 비교합니다.

핵심 포인트

  • 에이전트를 가르는 결정적 축은 '권한(어디까지 알아서 하게 둘 것인가)'이며, 웹 에이전트 < 코딩 에이전트/컴퓨터 유즈 < 자율 에이전트 순으로 자유도가 커짐
  • Claude Code는 2025년부터 운영된 기준점 도구로, Sonnet 4.6·Opus 4.8(2026년 5월 28일 출시) 기반이며 1M 컨텍스트를 정식 지원
  • Codex는 GPT-5.5 기반으로 가볍고 오픈소스이며, 권한 질문을 덜 하는 목적지향적 특성이 강점
  • Claude Cowork는 비개발자용 GUI 버전으로, 'Ask before acting' 방식으로 실행 전 계획을 보여주고 승인을 기다림
  • 코딩 에이전트는 이제 코드뿐 아니라 파일 정리·문서 자동화·앱 실행 등 컴퓨터의 거의 모든 작업으로 영역이 확장됨

왜 지금

코딩 에이전트가 '신기하다'와 '매일 쓴다'의 경계를 이미 넘어섰다는 진단과 함께, 시장에 난립한 에이전트 제품군을 구조적으로 정리해주는 콘텐츠가 필요해진 시점을 반영합니다.

실무 시사점

팀에 에이전트 도구를 도입할 때 '웹/코딩/자율' 중 어느 단계의 권한이 필요한 업무인지 먼저 구분하면, 과도한 자율성으로 인한 리스크나 반대로 불필요한 승인 병목을 피하는 데 도움이 됩니다.

실무 적용 포인트

  • 새 AI 도구를 검토할 때 컨텍스트·도구·권한·트리거 4가지 기준으로 먼저 분류해 업무 적합도를 판단한다
  • 비개발자 조직이라면 코딩 에이전트 대신 Claude Cowork 같은 GUI 기반 대안부터 시범 도입한다

실무 적용 포인트
